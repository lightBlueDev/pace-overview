# Sample Closure Checklist

This example shows the public-facing shape of a closure checklist used to confirm that a slice is actually complete.

## Sanitized Example

| Check | Status | What it means |
| --- | --- | --- |
| behavior updated | complete | the intended change exists in the relevant artifact or code surface |
| verification reviewed | complete | the work was inspected, tested, or otherwise checked appropriately |
| docs aligned | complete | surrounding documentation matches the implemented state |
| project status updated | complete | active status surfaces reflect reality |
| handoff ready | complete | the next session can orient without reconstructing context |
| residual risks recorded | complete | known limitations or deferred work are documented explicitly |

## Interpretation

PACE treats closure as agreement across multiple surfaces, not just a passing implementation detail.

A slice is more honestly complete when:

- the artifact exists
- the project state agrees that it exists
- the next operator can tell what changed
- any remaining uncertainty is explicit

That is stricter than "the work seems done," and that is the point.
